RFC: Service Connectivity Framework for SageMaker Endpoints

Overview

Our SageMaker platform hosts machine learning endpoints for multiple independent projects within a shared AWS account and shared VPC. While projects are logically separated through endpoint naming conventions, IAM permissions, and deployment governance, there is currently no standardized mechanism for managing connectivity from SageMaker endpoints to AWS managed services and internal platform services.

As additional platform capabilities are introduced, endpoints increasingly require access to services such as Amazon S3, AWS Secrets Manager, Feature Store, Hosted MLflow, OTLP Ingestion Service, and other internal APIs. Today, configuring these connections requires users to understand AWS networking concepts such as VPC Endpoints, Security Groups, Subnets, and IAM Roles.

This RFC proposes a Service Connectivity Framework that abstracts these implementation details behind platform-managed resources.

The framework introduces two new platform concepts:

* Service Connection – Defines how to securely connect to an AWS service or internal platform service.
* Network Access Profile – A reusable collection of Service Connections that can be granted to projects.

The framework separates service access into three independent layers:

* Network Connectivity – Can the endpoint establish a network connection?
* Authentication – Who is making the request?
* Authorization – Is the caller permitted to perform the requested operation?

This separation enables consistent governance while simplifying the deployment experience for end users.

⸻

Value Proposition

The proposed framework provides the following benefits:

* Standardized service onboarding across the platform.
* Simplified endpoint deployment by hiding AWS networking complexity.
* Project-level governance over service connectivity.
* Consistent application of least-privilege security.
* Reusable connectivity definitions shared across projects.
* Clear separation between network policy and access control.
* Reduced operational overhead as additional platform services are introduced.

⸻

Goals

* Introduce reusable Service Connections for AWS managed services and internal platform services.
* Introduce reusable Network Access Profiles.
* Eliminate the need for users to configure Security Groups, Subnets, IAM Roles, or VPC Endpoints.
* Provide project-level control over available services.
* Standardize the process for onboarding new platform services.
* Support private connectivity within the existing shared VPC architecture.
* Maintain compatibility with the existing SageMaker deployment workflow.

⸻

Non-goals

This proposal does not:

* Replace IAM authorization.
* Replace service-specific authorization models.
* Replace SageMaker networking primitives.
* Introduce dedicated VPCs for each project.
* Introduce service mesh technology.
* Modify SageMaker endpoint invocation authorization.
* Support arbitrary user-defined networking configurations.

⸻

Service Connection

A Service Connection is a reusable platform resource describing how to consume a service.

Each Service Connection contains:

* Service name
* Service type (AWS or Internal)
* Network connectivity configuration
* Authentication mechanism
* Authorization model
* Required infrastructure

Examples include:

* Amazon S3
* AWS Secrets Manager
* AWS STS
* AWS KMS
* Feature Store
* Hosted MLflow
* OTLP Ingestion Service

AWS Service Example

For Amazon S3, the Service Connection includes:

Network Connectivity

* S3 Gateway VPC Endpoint
* Required route tables
* Client Security Group
* Private connectivity

Authentication

* SageMaker Execution Role

Authorization

* IAM Policies
* Bucket Policies

⸻

Network Access Profile

A Network Access Profile is a reusable collection of Service Connections.

Examples:

aws-services

* Amazon S3
* Secrets Manager
* STS
* KMS

feature-store

* Amazon S3
* Feature Store
* Metadata Service

observability

* Amazon S3
* Hosted MLflow
* OTLP Ingestion Service

Network Access Profiles contain no project-specific configuration.

⸻

Project Authorization

Projects are granted permission to use one or more Network Access Profiles.

Example:

Project: Fraud

Allowed Profiles:

* aws-services
* feature-store

When deploying an endpoint, the Deployment Service validates that the requested profile is authorized for the project.

⸻

Creating a Service Connection

Step 1 – Configure Network Connectivity

Platform administrators provision the networking required by the service.

AWS Services

Examples:

* Gateway VPC Endpoints
* Interface VPC Endpoints
* Route Tables
* Security Groups

Internal Services

Examples:

* Security Groups
* ALB/NLB
* Private DNS
* AWS PrivateLink
* VPC Lattice (future)

The resulting connectivity information becomes part of the Service Connection definition.

⸻

Step 2 – Configure Access Control

Each service defines its authentication and authorization mechanisms.

AWS Services

Authentication

* SageMaker Execution Role

Authorization

* IAM Policies
* Resource Policies

Internal Services

Authentication

* IAM
* JWT
* mTLS (future)

Authorization

* Project ACLs
* Service-specific RBAC

⸻

Step 3 – Publish the Service Connection

Once networking and authorization have been validated, the Service Connection is published.

Published Service Connections become available for inclusion in Network Access Profiles.

⸻

Granting Access

Granting access consists of two administrative operations.

Grant Network Connectivity

The Platform Administrator grants a Network Access Profile to a project.

Example:

Project

Fraud

Granted Profile

feature-store

This authorizes future SageMaker endpoints within the project to use the profile.

⸻

Grant Access Control

The service owner grants the project access to the service.

Example (Amazon S3)

IAM Policy

Allow:

* s3:GetObject
* s3:PutObject

Bucket

fraud-data

Principal

sagemaker-exec-fraud

Network connectivity alone does not provide authorization.

⸻

Endpoint Deployment

Deployment request:

Project:

Fraud

Profile:

feature-store

Deployment Service performs:

1. Validate project.
2. Validate Network Access Profile.
3. Resolve Service Connections.
4. Resolve Security Groups.
5. Resolve Subnets.
6. Resolve Execution Role.
7. Generate CloudFormation.
8. Deploy SageMaker endpoint.

The deployment service owns all AWS networking resources.

⸻

Runtime Flow

SageMaker Endpoint
        │
Network Connectivity
        │
Authentication
        │
Authorization
        │
Target Service

Every request must satisfy all three layers.

⸻

Amazon S3 Example

Project requests:

Network Profile

aws-services

Deployment Service provisions:

* VPC Configuration
* Security Groups
* S3 Gateway Endpoint
* SageMaker Execution Role

Runtime:

1. Endpoint connects to Amazon S3 through the Gateway VPC Endpoint.
2. AWS authenticates the request using the SageMaker Execution Role.
3. Amazon S3 evaluates IAM policies.
4. Amazon S3 evaluates the Bucket Policy.
5. Access is granted or denied.

This demonstrates the separation between:

* Network Connectivity
* Authentication
* Authorization

Risks

Risk	Mitigation
Too many Network Access Profiles	Maintain a curated catalog of reusable profiles.
Duplicate service definitions	Treat Service Connections as reusable platform resources.
Excessive permissions	Separate network connectivity from authorization and follow least-privilege principles.
Configuration drift	Manage all Service Connections and Network Access Profiles through Infrastructure as Code.
Future networking changes	Hide implementation details behind Service Connections so the deployment interface remains stable.

⸻

Alternative Solutions

Users Configure AWS Networking

Allow users to specify Security Groups, Subnets, IAM Roles, and VPC Endpoints during deployment.

Pros

* Maximum flexibility.

Cons

* Difficult to govern.
* Error-prone.
* Exposes AWS implementation details.
* Poor user experience.

⸻

Per-Project VPC

Deploy each project into its own VPC.

Pros

* Strong network isolation.

Cons

* Higher operational overhead.
* Increased infrastructure cost.
* Reduced resource sharing.
* More complex routing.

⸻

Continue Current Model

Continue using manually configured networking.

Pros

* No implementation effort.

Cons

* Inconsistent networking.
* Difficult governance.
* Poor scalability.
* Increased operational burden.

I think this version is closer to an architecture RFC than the previous drafts because it clearly distinguishes the three platform resources (Service Connections, Network Access Profiles, and Project Grants) and includes a complete lifecycle from service onboarding through deployment and runtime authorization. It’s also generic enough that the same framework can later be used for ECS services or AgentCore workloads, not just SageMaker endpoints.
